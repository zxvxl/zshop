# ZShop — Next.js 14 Digital Goods Store with USDT Payment

A self-service digital goods store (card-key marketplace) built on Next.js 14 App Router with SQLite/Prisma, BSC (BEP-20) USDT payment detection via BscScan API polling, and a basic admin panel protected by HTTP Basic Auth. The architecture is straightforward: customers place orders, the system generates a unique USDT amount per order, a cron job polls BscScan for matching incoming transfers, and cards are auto-delivered on match.

**Watch for:**
- **Cron container uses `require("dotenv")` but no `dotenv` dependency** (confirmed) — `cron.js` will crash on startup in the Docker cron container.
- **Missing `package-lock.json` breaks Docker build** (confirmed) — the Dockerfile uses `npm ci` which requires a lockfile. Build will fail.
- **Payment amount collision allows order hijacking** (confirmed) — any transfer matching the unique amount pays the first matching pending order, regardless of sender. Combined with the predictable amount-generation scheme, an attacker can front-run legitimate payments.
- **Quantity not multiplied into payment amount** (confirmed) — ordering N items generates an amount based on single-unit price, so the customer pays for 1 but receives N cards.
- **`generateUniqueAmount` can recurse infinitely** (likely) — if all amount slots in a 30-minute window are exhausted (high-volume scenario), the function recurses with no base case or depth limit.

---

## High-level view

The project is a well-structured single-repo Next.js 14 App Router application with clear separation between public shop routes (under `(shop)` route group) and admin routes. The data layer uses Prisma with SQLite, which is appropriate for a low-to-medium-volume card store deployed as a single Docker service.

The payment flow relies on amount-based matching: each order gets a unique USDT amount (base price + incrementing suffix), and a cron job polls BscScan every 5 seconds for incoming transactions. This design avoids webhook complexity but introduces a critical trust gap — there's no sender verification, so anyone who transfers the right amount to the wallet address gets credit for the order. Additionally, the amount generation ignores the `quantity` parameter, creating a revenue-loss bug for multi-quantity orders.

The Docker setup has multiple runtime gaps: `cron.js` depends on `dotenv` (not in `package.json`), the standalone output won't bundle `node-cron`, and there's no `package-lock.json` for `npm ci`. These are all build/startup-blocking issues.

The i18n setup uses `next-intl` v3.22+ with a cookie-based locale. The plugin auto-discovers `./i18n/request.ts` (no path argument needed in v3.22+), but `cookies()` is called without `await` which will break in Next.js 15 or newer 14.x patches.

5. **Payment amount collision / order hijacking** — any wallet sending the matching USDT amount gets the order fulfilled. No sender verification. An attacker can observe the predictable amount pattern and front-run.
6. **`generateUniqueAmount` unbounded recursion** — if collision detection keeps finding existing orders, the function recurses without a depth limit, risking stack overflow.
7. **Amount matching uses string equality on floating-point-derived values** — `BigNumber.toString()` output and `product.price.toFixed(2) + no` may produce different decimal representations for the same value, causing missed payment matches.
8. **Default admin credentials `admin:123456`** — the fallback in middleware is trivially guessable. No enforcement to change it, no rate limiting on auth attempts.
9. **Order query endpoint leaks card content without authentication** — `/api/order/query` returns card keys to anyone who knows an email address. No rate limiting, no auth.
10. **`/api/order/check` endpoint has no authentication** — anyone can poll any order by `orderNo` and see status, address, email, and card content.
11. **Race condition in card delivery** — `deliverCards` reads available cards and updates them one-by-one without a transaction. Concurrent cron executions can assign the same card to multiple orders.
12. **`cookies()` must be awaited in newer Next.js 14** — `i18n/request.ts` calls `cookies()` without `await`, which breaks in Next.js 14.2.x+ and 15.x.
13. **Revenue calculation loads all paid orders into memory** — `AdminDashboard` fetches every paid order to sum amounts client-side. Should use Prisma aggregate.

## Build-blocking: missing lockfile and cron dependencies

The Dockerfile starts with `npm ci`, which hard-requires a `package-lock.json`:

```dockerfile
RUN npm ci
```

No lockfile exists in the repository (not present, not gitignored). The Docker build will fail immediately with `npm ci can only install packages when your package.json and package-lock.json are in sync`.

`cron.js` has two additional dependency problems:

```js
const cron = require("node-cron");
require("dotenv").config();
```

`dotenv` is not in `package.json` at all — `MODULE_NOT_FOUND` on startup. `node-cron` is in dependencies but the Dockerfile uses Next.js standalone output (`output: 'standalone'`). Standalone mode only bundles dependencies traced through Next.js page/route imports. Since `cron.js` is a separate entrypoint never imported by Next.js, `node-cron` won't be in the standalone `node_modules`. The cron container copies from the standalone build and will fail to find it.

## Payment matching — trust model and revenue-loss bug

The payment flow generates a unique USDT amount per order and matches incoming blockchain transactions by amount alone:

```ts
const order = await prisma.order.findFirst({
  where: {
    amount,
    status: "pending",
    createdAt: { gt: dayjs().subtract(30, "minute").toDate() },
  },
});
```

No sender verification exists. Any address that sends the matching amount to the wallet fulfills the order. The amount generation is predictable (sequential suffix per product), so an attacker who sees one payment amount can predict the next.

More critically, `generateUniqueAmount` ignores the `quantity` parameter entirely:

```ts
const amount = `${product.price.toFixed(2)}${no}`;
```

For a $25 product ordered with quantity=3, the amount is `25.001` — the customer pays $25 and receives 3 cards. The `BuyForm` UI shows `price * quantity` as the total, but the actual payment amount sent to the blockchain matching system is just the unit price. This is a confirmed revenue-loss bug unless the intent is one-payment-per-card (but the UI contradicts that interpretation).

The recursion in `generateUniqueAmount` has no termination guard:

```ts
if (existing) {
  return generateUniqueAmount(product); // no depth counter
}
```

Under high order volume for a single product, or adversarial conditions where an attacker creates many orders to exhaust amount slots, this recurses until stack overflow.

## Card delivery race condition

```ts
const cards = await prisma.card.findMany({
  where: { productId: order.productId, sold: false },
  take: order.quantity,
});

for (const card of cards) {
  await prisma.card.update({
    where: { id: card.id },
    data: { sold: true, orderId: order.id },
  });
}
```

No transaction wraps this read-then-write sequence. The cron job runs every 5 seconds and processes all transactions in a loop. If BscScan returns the same transaction hash before the first processing completes (network delay, retry), two `checkTx` calls fire concurrently. While `existingTx?.matched` provides some guard, the window between `findFirst` and the `update` is still open. The `deliverCards` function's own guard (`if (order.cards.length > 0) return`) checks after the fact but the actual card assignment loop has no atomicity guarantee. Two concurrent loops can both read the same unsold cards and attempt to assign them.

Fix: wrap the card assignment in `prisma.$transaction()` with a serializable isolation level, or use `updateMany` with a `sold: false` condition to atomically claim cards.

## Data exposure through public endpoints

`/api/order/query` accepts an email address and returns all orders with card content for paid orders:

```ts
const orders = await prisma.order.findMany({
  where: { OR: [{ orderNo: trimmed }, { email: trimmed }] },
  include: { cards: { select: { content: true } } },
});
```

Card content is the product itself (account credentials). Any attacker who knows or guesses a customer email (common — emails are often predictable) can retrieve all their purchased credentials. This endpoint has no rate limiting and no authentication. The minimum 3-character keyword length is the only gate.

`/api/order/check` is used by the payment status poller and also returns card content for any order number. The 10-character nanoid provides ~56 bits of entropy (50 chars, 10 positions: log2(50^10) ≈ 56), which is resistant to random brute force but the endpoint still leaks `email` and `address` fields unconditionally.

## Admin credentials

```ts
const [AUTH_USER, AUTH_PASS] = (process.env.HTTP_BASIC_AUTH || "admin:123456").split(":");
```

The fallback `admin:123456` means any deployment where `HTTP_BASIC_AUTH` isn't set (including running `npm run dev` locally without an `.env` file) is fully accessible with trivial credentials. HTTP Basic Auth also sends credentials in base64 (not encrypted) on every request, which is acceptable only over HTTPS. No TLS enforcement exists in the app itself (would need to be handled at the reverse proxy level).
