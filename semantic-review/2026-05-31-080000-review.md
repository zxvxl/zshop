# Multi-provider payment system with USDT and Alipay

This change replaces the hardcoded BSC wallet payment flow with a pluggable payment channel system. Channels are stored in a new `PaymentChannel` table, each with a provider type (`usdt_bsc`, `alipay_face`, `alipay_web`) and JSON config. Order creation now resolves a channel, delegates to the appropriate provider for payment details, and stores a `channelId` on the order. The cron job iterates channels instead of reading from env vars. Alipay payments land via a notify callback at `/api/payment/notify/alipay`. The frontend was refactored to use a shared `apiPost` wrapper that unwraps the `{code, data, message}` envelope.

Watch for: the payment status page only renders BSC address mode — Alipay QR/URL orders will show a blank or stale address field (confirmed). Server actions in the admin payment page have no independent auth check and can be invoked directly, bypassing middleware (confirmed). The Alipay notify handler doesn't validate that the paid amount matches the order amount, allowing underpayment attacks (confirmed).

## High-level view

The payment provider abstraction (`PaymentProvider` interface with `createPayment`, `checkPayment`, `handleNotify`) is clean and the registry pattern in `lib/payment/index.ts` makes adding new providers straightforward. The channel model stores config as a JSON string, which works for SQLite but means no schema validation happens at the database level — malformed config will surface as runtime errors inside providers.

The order creation flow correctly falls back to env-based USDT when no channels are configured, preserving backward compatibility. However, the response shape from order creation now includes a `payment` object (with mode, payUrl, qrCode, address) that the buy-form discards — it just reads `order.orderNo` and redirects to the payment page, which only knows how to display a BSC address.

The Alipay callback handler verifies the signature via `sdk.checkNotifySign`, but treats signature verification as the only gate — it doesn't confirm that the amount Alipay reports matches the order's expected amount, so a signature-valid notification for a smaller amount would still trigger delivery.

The admin page uses Next.js server actions for channel CRUD, which are convenient but inherit a known gap: server actions are separate HTTP endpoints that don't pass through the route middleware matcher. An attacker who discovers the action endpoint can invoke `createChannel` or `deleteChannel` without authentication.

<summary>Issues (6)</summary>

1. **Payment page can't render Alipay orders** — `payment-status.tsx` only shows address/amount for BSC USDT. Orders paid via Alipay (mode `qrcode` or `url`) have no payUrl/qrCode rendered; users see a blank payment instruction. Add conditional rendering for `payUrl` redirect and `qrCode` display.
2. **Alipay notify skips amount validation** — The callback verifies the Alipay signature but doesn't compare `body.total_amount` against `order.amount`. A valid notification for a partial payment would still trigger card delivery. Add `if (result.amount !== order.amount) return fail`.
3. **Server actions unprotected by auth** — `createChannel`, `toggleChannel`, `deleteChannel` in the admin payment page are server actions callable via direct POST without passing through the admin middleware. Add an auth check (e.g., `getAuthUser()` with role verification) at the top of each action.
4. **USDT amount-matching race window** — The single-address model matches orders by exact amount string. If a transaction for `10.002` arrives but was intended for a different order (`10.001`), the system matches it to the wrong order. No per-order deposit address exists to disambiguate. Worth documenting for operators.
5. **Payment channel config stored in plaintext** — Private keys (Alipay `privateKey`) are stored as plaintext in the `config` JSON column. Anyone with database read access sees raw credentials. Consider encrypting sensitive fields at rest.
6. **Cron job skips Alipay polling** — The job route filters `channel.provider.startsWith("usdt_")`, so Alipay orders that miss the callback are never recovered. Since `AlipayProvider.checkPayment` exists and works, consider including Alipay channels in the polling loop as a fallback.

## Payment status page only renders BSC address mode

The `payment-status.tsx` component accepts an `OrderData` interface with a fixed `address` field and renders it as a BSC wallet address with a copy button. It has no concept of `payUrl` (redirect to Alipay cashier) or `qrCode` (QR image for face-to-face payment).

When `order/route.ts` creates an Alipay order, it stores `payUrl` or `qrCode` on the order record. But the buy-form only reads `order.orderNo` and redirects to `/pay/[orderNo]`. The pay page server component reads the order from the database and passes `order.address` to `PaymentStatus` — for Alipay orders, `order.address` is still the env-based BSC wallet (set by `createOrder` in `lib/order.ts`: `address: process.env.BSC_WALLET || ""`), which is meaningless in an Alipay context.

The fix requires: (1) passing `payUrl`/`qrCode` from the order record into the `PaymentStatus` props, and (2) rendering different UIs based on the payment mode — a redirect button for `payUrl`, a QR image for `qrCode`, and the current address display for USDT.

## Alipay notify callback doesn't validate payment amount

In `app/api/payment/notify/alipay/route.ts`, after signature verification succeeds, the handler marks the order as paid:

```typescript
const result = await provider.handleNotify(params, config);
if (!result.success) return new Response("fail", { status: 400 });

// Mark order as paid — no amount comparison
await prisma.order.update({
  where: { id: order.id },
  data: { status: "paid", paidAt: new Date() },
});
await deliverCards(order.id);
```

The `handleNotify` method returns `{ success: true, amount: body.total_amount }` but the caller never checks `result.amount` against `order.amount`. A valid notification for a partial payment would trigger card delivery.

The fix is one line: `if (result.amount !== order.amount) return new Response("fail", { status: 400 });` before the update.

## Server actions bypass middleware auth

Next.js server actions are compiled into standalone API endpoints. The middleware matcher `["/admin/:path*"]` only protects page navigation routes, not the internal action endpoints. This means:

```
POST /_next/action/... (createChannel action)
```

...can be called by any unauthenticated client who discovers or brute-forces the action ID. Since `createChannel` takes a `FormData` with provider config (including private keys in plaintext), this is a data-write vulnerability.

Each server action should start with:
```typescript
const { cookies } = await import("next/headers");
// verify JWT from cookie, check role === "admin"
```

## alipay-sdk CJS interop

`alipay-sdk@^4.0.0` uses CommonJS internally. The dynamic import pattern in `AlipayProvider` (`await import("alipay-sdk")`) with the `.default` accessor is the standard interop pattern and should work. However, if the package moves to ESM in a future minor version (unlikely for v4 but possible), the `.default` access would break. Pinning to `~4.0.0` instead of `^4.0.0` would guard against this (possible).
